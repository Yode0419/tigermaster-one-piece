# 知識庫工作流擴展：Wireframe 生成能力

## 概述

- **類型**：知識庫的能力擴展規劃
- **受影響角色**：設計師、工程師（知識庫的使用者）

---

## 問題定義

### 痛點

設計師在進行功能設計時，討論過程缺乏即時的視覺化介面預覽。從設計討論到 Figma 設計稿之間存在一段落差，需要從零手動建立 wireframe，效率低且容易與討論內容脫節。

### 現況

知識庫目前以文字形式記錄產品邏輯與流程，尚未建立設計探索的工作流。`/luffy` 技能已存在但尚未在實際設計流程中跑起來。Wireframe 生成能力目前不存在。

### 成功標準

設計師在設計討論的任意階段，可以呼叫一個技能生成可瀏覽的介面預覽，該預覽已反映既有產品的 UI pattern。設計師可用來即興預覽、確認方案，或比較不同 UI 方向，再決定是否進入 Figma 精修或直接交付開發。

---

## 設計方向

**技能的三種使用情境：**

| 情境 | 觸發時機 | 預期輸出 |
|---|---|---|
| 即興預覽 | 討論進行中，想看某個概念的畫面 | 對應概念的基礎介面預覽 |
| 方案確認 | 設計方向定了，需要較完整的畫面呈現 | 反映完整設計決策的 prototype |
| 方案比較 | 同一需求有多個 UI 方向，需要對照選擇 | 多個 UI 變體，可逐一預覽比較 |

**工作流（目標狀態）：**

```
/luffy 設計討論（任意階段）
  → 想預覽介面時 → 呼叫 wireframe 技能（隨時可用，不綁定 PRD）
    → 介面預覽（參考既有產品 UI pattern）
      → 繼續討論 / 嘗試不同方案 / 調整後進入開發
```

**核心概念：**

- 技能不綁定 PRD，輸入可以是任意描述（功能概念、頁面改動、設計片段）
- 介面預覽應反映既有產品的 UI pattern，而非從零生成陌生的樣式
- 既有產品 UI pattern 需以某種形式存放於知識庫，供技能讀取參考
- Design system 規範維持在 Figma，不複製進這個知識庫

## 影響範圍

**需要新增的內容：**

| 項目 | 說明 | 狀態 |
|---|---|---|
| UI pattern 參考文件 | 既有 app 與後台的常用版型與 UI 慣例，供 wireframe 技能讀取 | 待建立（路徑與格式 TBD）|
| Wireframe 生成技能 | 讀取設計描述 + UI pattern，生成介面預覽 | 待建立（用 `/skill-creator`）|

**三個專案的分工（考慮方向，尚未確認）：**

| 專案 | 內容 | 主要使用者 |
|---|---|---|
| 這個庫（產品知識） | 產品邏輯、流程、決策、UI patterns | 所有人 |
| Design system 庫（未來） | Figma 規範、視覺語言 | 設計師、前端 |
| 程式碼庫 | 實際元件、實作細節 | 工程師 |

## 決策記錄

**Design system 不納入這個知識庫**
→ 設計規範有自己的迭代節奏，混入會增加維護成本。知識庫只存 UI pattern 描述（輕量、給 AI 讀的格式），完整規範維持在 Figma。

**技能不綁定 PRD，隨時可呼叫**
→ 視覺化預覽是設計探索的工具，而非流程終點；綁定 PRD 會限制使用彈性，無法支援即興預覽與方案比較的情境。
