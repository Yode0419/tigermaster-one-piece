# 編號規則

## 概述

說明平台各類識別碼的格式規則，包含訂單編號與用戶 ID 的命名慣例，以及其他資料表的關聯設計原則。

---

## 背景與脈絡

平台採用語義化編號設計訂單與用戶的識別碼，讓客服、工程師在查案或對帳時能直接從編號判讀日期與類型。其他資料表（如報價單、交易紀錄）則使用資料庫自動遞增流水 ID。

---

## 訂單編號（Order ID）

| 欄位 | 說明 |
|------|------|
| 前綴 | `RO`（Repair Order） |
| 日期段 | `YYYYMMDD`（訂單建立日期） |
| 流水號 | 五位數，補零，每天從 `00001` 重置 |
| 計數範圍 | 全平台統一，每日重置 |

**範例**：`RO2026041300009`

---

## 用戶 ID（User ID）

| 欄位 | 說明 |
|------|------|
| 前綴 | 無（純數字） |
| 日期段 | `YYYYMMDD`（用戶註冊日期） |
| 流水號 | 五位數，補零，每天從 `00001` 重置 |
| 計數範圍 | 客戶與師傅共用同一序列（角色為用戶的屬性層，不影響 ID 格式） |

**範例**：`2024090200007`

---

## 其他資料表編號慣例

平台其他資料表（如報價單、交易紀錄等）一律使用資料庫自動遞增流水 ID，不含日期或語義前綴。各表之間透過關聯欄位串接，實際串接邏輯依資料表而異。

### 第三方金流編號

- **發票號碼**：由 ezpay 自動產生，平台不另設格式
- **藍新刷卡交易**：平台將訂單編號回傳給藍新作為對帳依據
