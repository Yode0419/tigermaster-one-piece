# 訂單旗標：不支付給師傅車馬費

## 概述

- **類型**：既有功能優化（補完開發到一半的旗標邏輯）
- **受影響角色**：超級管理員（後台操作者）、師傅（收款受影響）

---

## 問題定義

### 痛點

後台訂單旗標區塊中的「不支付派遣費」checkbox 原本沒有任何實際邏輯，只是一個純開關。若要取消師傅車馬費，需手動進入師傅詳情頁編輯收入金額並自行填寫註記，操作繁瑣。老闆希望能在訂單詳情頁一鍵完成。

### 現況

依師傅管理辦法第二之四，若師傅未於約定日期準時到場且未取得客戶同意，公司可依「派遣費用制度」取消師傅的車馬費。但目前系統無法自動執行此邏輯，須人工至師傅詳情頁的收入列表手動編輯金額，操作繁瑣、無系統化的操作記錄。

### 成功標準

超級管理員可直接在訂單詳情頁勾選 / 取消旗標，系統自動更新師傅車馬費收入記錄並寫入系統註記，無需跨頁操作。

---

## 設計方向

將旗標文案從「不支付派遣費」改為「不支付給師傅車馬費」，並補上完整的業務邏輯。旗標的作用分為兩個時機：

### 權限控制

「不支付給師傅車馬費」旗標的操作與儲存限**超級管理員**才可使用。其他後台使用者可見此旗標，但無法操作。

### 儲存旗標時

```
有車馬費記錄？
├─ 是
│   ├─ 勾選      → 改為 $0 ＋ 寫系統註記
│   └─ 取消勾選  → 改為 $200 ＋ 寫系統註記
└─ 否
    訂單已終止（狀態 100）？
    ├─ 是
    │   ├─ 勾選      → 僅儲存旗標
    │   └─ 取消勾選  → 補建 $200 收入記錄 ＋ 寫系統註記
    └─ 否            → 僅儲存旗標，等待車馬費產生時機
```

### 車馬費產生時

在既有的車馬費產生邏輯執行時，加入旗標判斷：

```
旗標勾選？
├─ 是  → 不產生車馬費
└─ 否  → 依原有邏輯產生車馬費收入記錄
```

### 系統註記格式

每次旗標操作寫入一行至師傅收入記錄的「註記」欄位，多次操作逐行追加，不覆蓋舊紀錄：

```
YYYY/MM/DD HH:MM [帳號ID] 勾選「不支付給師傅車馬費」旗標
YYYY/MM/DD HH:MM [帳號ID] 取消「不支付給師傅車馬費」旗標
```

---

## 影響範圍

**師傅收入觸發規則（背景）**

| 訂單狀態 | 說明 | 師傅收入 |
|---------|------|---------|
| 狀態 10–25 終止 | 師傅未完成場勘 | 不產生任何收入 |
| 狀態 30–45 終止 | 場勘完成，報價未同意 | 產生 $200 車馬費（類型：訂單取消） |
| 正常完工（狀態 105） | 報價同意並驗收完成 | 產生報價金額收入（類型：完成訂單） |

**受影響系統**
- 訂單旗標操作權限控制
- 訂單旗標儲存邏輯
- 師傅收入記錄建立 / 更新邏輯
- 車馬費產生流程（既有邏輯新增旗標判斷）

---

## 決策記錄

**旗標不做 disabled**
→ 旗標在不適用的情境下仍保持可操作，不根據訂單狀態動態 disabled。
→ 原因：操作者熟悉業務情境，不適用時操作無副作用；disabled 的判斷邏輯會增加開發成本與維護負擔。
→ 放棄替代方案：根據狀態動態 disabled 旗標。

**使用師傅收入記錄作為判斷依據**
→ 儲存旗標時的立即觸發以「有無車馬費收入記錄」為主要判斷條件，而非查詢場勘狀態或報價單。
→ 原因：師傅收入表單一查詢即可隱含多個條件，減少跨資料表查詢複雜度。

**取消旗標時固定恢復 $200**
→ 取消旗標時一律將車馬費恢復為 $200。
→ 原因：此旗標的主要使用情境為訂單在狀態 30–45 取消，車馬費依制度為 $200。極端情況（如客戶已同意報價但手動關閉訂單）由管理員事後手動修正，不在此邏輯處理範圍內。
