# 撥款作業介面調整討論紀錄

## 核心目標

介面與文案應讓會計自然理解操作流程：

**查看師傅收入明細 → 加入匯款清單 → 去銀行匯款 → 回系統確認 → 完成撥款**

---

## 流程導覽列

- **位置**：頁面頂部，橫跨三個區塊上方
- **形式**：純展示 banner，視覺化呈現上方操作流程文字
- **互動**：無（非可點擊步驟）
- **目的**：幫助新手進頁面即建立操作全局認知

---

## 第一區塊（師傅收入明細）

### 文案與標籤

| 位置              | 原有         | 確認             | 設計想法                                                                   |
| ----------------- | ------------ | ---------------- | -------------------------------------------------------------------------- |
| 區塊標題          | 匯款資訊     | **師傅收入明細** | 原語意模糊，明確說明此區是收入資料的查閱視角                               |
| Dropdown 標籤     | 匯款週期     | **結算週期**     | 對應平台結算機制，與第三區塊「實際匯款週期」區隔                           |
| 表格欄位          | 預計匯出週期 | **結算週期**     | 與 dropdown 標籤一致，避免同一欄位出現不同名稱                             |
| 欄標題            | 加入匯款紀錄 | **加入匯款清單** | 對應第三區塊標題，使動作目的地清楚                                         |
| 按鈕 A            | 預設         | **全額加入**     | 原「預設」語意不清，明確說明動作與金額類型；直接加入，不開 modal           |
| 按鈕 B            | 自選         | **部分加入**     | 與「全額加入」形成對比；開 modal 輸入本次金額                              |

**刪除毛利(%)欄位**：算法有誤、資訊有雜訊；獲利欄已有負值紅字提示，可取代異常偵測需求。同時減少欄位數量，避免畫面超出可視範圍。

### 版面

- **結算週期 dropdown + 總數 badge + 下載（靠右）**：badge 靠近控制它的 dropdown；下載為低頻輔助動作，靠右對齊
- **欄標題「加入匯款清單」旁加 `?` icon（hover tooltip）**：說明全額加入與部分加入的差異，文案待優化

### 互動行為

- **可查看任意結算週期**：此區塊定位為收入資料總覽，兼顧當期操作與歷史查閱下載
- **加入後（全額或部分）**：該行變暗，左側顯示「已加入 / 匯款清單」綠色文字，右側顯示「取消加入」outlined button；「取消加入」為就地反向撤銷，避免加錯時需至第三區塊操作
- **第三區塊「移除」後同步恢復**：兩區塊文案職責分明，狀態保持同步，防止行永久暗掉但清單裡找不到的孤立情況
- **確認匯款後（全額）**：付款狀態變「已匯款」，行恢復正常顯示，加入按鈕消失；已匯款不應再被重複加入
- **確認匯款後（部分）**：付款狀態變「部分匯款」，行恢復正常顯示，加入按鈕重新出現；師傅收入(餘)即時扣除已匯金額，全額／部分加入皆以新餘額為上限；允許多次部分匯款，餘額歸零時自動轉為「已匯款」

### 部分加入 Modal

原後台 modal：標題「加入匯款紀錄」＋金額輸入欄（預設 0）＋取消／確認，資訊過少。

調整方向（改動幅度最小）：

- **標題**：部分加入匯款清單
- **說明文字**：請輸入本次匯款金額，剩餘收入將保留至下次撥款
- **金額輸入**：預設空白（原為 0）
- **驗證**：不能為 0、不能超過師傅剩餘收入；inline 紅字提示，確認按鈕 disable
- **不在 modal 內顯示師傅剩餘收入**：會計點「部分加入」時，表格該行本已顯示剩餘金額，資訊足夠；在 modal 內顯示需帶入個別師傅資料，改動幅度過大

## 第二區塊（前期未匯）

跨期補漏區塊，自動彙整所有結算週期中付款狀態為「未匯款」或「部分匯款」的收入資料，供會計清查與補匯。

### 區塊順序決策紀錄

師傅收入明細排第一、前期未匯排第二，理由如下：

- 會計每次作業兩個區塊都必須處理，但操作順序沒有固定，傾向依金額大小與期別久遠做判斷（先消大金額、再削舊的）
- 兩個區塊對會計而言是**並列**關係，非先後依賴
- 本期資料量較多，以量較大的區塊作為視覺錨點較為合理
- 前期未匯定位為補漏清查，排第二符合主從邏輯

**待解決的設計問題（非本次範圍）**：會計需要跨兩個區塊比較金額與期別才能做判斷，但現行設計兩個區塊各自獨立，無法一眼綜覽所有待匯款項。此問題預計在未來「以師傅為單位彙整視角」中一併解決。

### 版面

- **欄位結構與第一區塊一致**
- **下載按鈕（靠右）**：低頻輔助動作
- **無結算週期 dropdown**：此區塊定位為全期跨期顯示，不需切換

### 排序

- 依**結算週期由舊到新**排序：越舊的欠款越靠前，形成自然催促感，避免久未處理的項目被新資料蓋過

### 互動行為

- 與第一區塊一致：加入後變暗、顯示「已加入 / 匯款清單」綠色文字、顯示「取消加入」按鈕
- 與第三區塊狀態保持同步（移除後恢復、確認匯款後更新付款狀態）

### 未來方向

- 改為以師傅為單位的彙整視角：同一師傅的多筆跨期收入合併顯示，減少人工加總

## 第三區塊（匯款清單）

### 文案與標籤

| 位置           | 原有     | 確認                             | 設計想法                                                                         |
| -------------- | -------- | -------------------------------- | -------------------------------------------------------------------------------- |
| 區塊標題       | 匯款紀錄 | **匯款清單**                     | 原「紀錄」偏歷史感；「清單」可同時容納待匯款與已匯款，且與「加入匯款清單」對齊 |
| 狀態「建立」   | 建立     | **待匯款**                       | 原為系統語言；「待匯款」清楚說明當前狀態                                         |
| 確認按鈕       | 確認匯款 | **確認已匯款**                   | 加上「已」字，明確說明這是事後回系統紀錄的動作，非「點了才匯」                   |
| 引導文字       | 無       | **請先完成銀行匯款，再點擊確認** | 提醒會計操作順序，避免尚未匯款就點確認                                           |

### 版面

- **匯款週期 dropdown + 下載（靠右）**：可切換查看各期匯款紀錄，結構與第一區塊一致
- **引導文字（左）+ 確認已匯款按鈕（右）**：引導文字撐左，主要操作按鈕靠右
- **表格欄位順序**：訂單編號 / 收款人姓名 / 類型 / 匯出款項 / 狀態 / 實際匯款週期 / 建立日期 / 移除

### 互動行為

- **確認已匯款加彈窗**：批次動作影響範圍大且不可逆；彈窗標題「請確認已在銀行完成匯款」，內文說明確認後無法移除；按鈕為「取消」與「確認已匯款」
- **批次確認**：一次確認當期所有待匯款項目
- **移除按鈕僅待匯款狀態顯示**：確認後按鈕消失，避免誤觸
- **已匯款歷史保留在區塊內**：以匯款週期 dropdown 切換查看，不另加 tab 或收合機制

### 未來方向

- 改為以師傅為單位的彙整視角：一位師傅的多筆收入合併顯示，讓會計直接看到應撥總金額，減少人工加總
