# 匯款清單後期優化：師傅分組視角

## 概述

- **類型**：既有功能優化
- **受影響角色**：客服（會計）
- **相關文件**：[撥款作業介面調整討論紀錄](payout-interface-redesign.md)、[對帳與撥款作業優化：主管討論結果](reconciliation-discussion-results.md)

---

## 問題定義

### 痛點

匯款清單目前以逐筆收入資料呈現，同一位師傅的多筆收入分散在清單中。會計在撥款日前往銀行匯款時，需要自行用 Excel 加總才能得知「這次要匯給每位師傅多少錢」。

此外，清單中無法直接看到每位師傅的請款方式，會計難以快速判斷當期是否需要對某位師傅的收入資料進行調整。

### 現況

- 匯款清單為單一平面表格，每頁最多 10 筆，每期約 50–80 位師傅、100–200 筆資料
- 會計需自行用 Excel 加總每位師傅的應匯金額
- 師傅請款方式無法在匯款清單中直接看到

### 成功標準

- 進入匯款清單後，不需要 Excel，直接看到每位師傅的應匯總金額與請款方式
- 若某筆收入匯款失敗，可將該筆從清單中移除，確認清單內皆為匯款成功的資料後，執行批次確認

---

## 設計方向

在第三區塊（匯款清單）加入**師傅分組邏輯**，讓同一師傅的多筆收入聚合為一個群組，顯示應匯總金額與請款方式。確認已匯款機制維持批次操作，不改動。

**範圍說明**：本次只調整第三區塊，第一、二區塊維持前期優化成果不變。師傅分組若未來驗證效果良好，可延伸至第一、二區塊，但非本次範圍。

### 明細表格

右側面板顯示選中師傅的個別收入筆數，欄位結構與現有匯款清單一致，每一筆各有「移除」按鈕（僅待匯款狀態顯示）。

### 匯款失敗的處理流程

若某筆收入匯款失敗，會計先將該筆資料從清單中移除。清單內剩餘資料即為本次匯款成功的項目，統一執行批次確認。

### 移除維持逐筆操作

不新增師傅層級的移除按鈕。理由：一位師傅每期收入筆數有限，逐筆移除操作快速，師傅層級批次移除效益不大。

---

## 介面結構

採用 **B｜側欄＋詳情**：左側師傅名單，右側顯示選中師傅的帳戶資訊、明細表格與狀態。

曾比較兩種方向：

| 方向 | 描述 |
|------|------|
| **A｜手風琴式** | 頁面垂直排列所有師傅 Header，點擊展開明細表格 |
| **B｜側欄＋詳情** | 左側師傅名單，右側顯示選中師傅的明細表格 |

選擇 B 的理由：
- 左欄持續顯示所有師傅總覽，不需任何操作即可掌握全局
- 展開明細時版面不位移，視線焦點穩定
- 右側空間充裕，明細表格閱讀更清楚
- 處理單一師傅時，仍可透過左欄掌握其他師傅的狀況

### 左側欄內容

每位師傅一行，顯示：

- **師傅姓名**（主要識別）
- **請款方式**（小標籤形式：開立發票 / 整筆個人 / 分期個人）
- **該期總金額**（待匯款＋已匯款的所有記錄加總）

頂部加入**姓名搜尋欄**，即時過濾師傅清單。每行採高密度設計，確保 50–80 位師傅不需過度捲動。

**金額顯示說明**：左欄金額為該師傅本期所有記錄的總和，不區分狀態。待匯款與已匯款的細節由右側明細表格呈現。

### 右側 Header 內容

顯示選中師傅的：師傅姓名 ｜ 請款方式 ｜ 銀行帳戶資訊（銀行名稱＋帳號）｜ 該期總金額

## 決策記錄

**確認已匯款維持批次操作，不改為逐師傅**
→ 匯款失敗的處理方式是「移除該筆後再批次確認」，清單內剩餘資料皆為當次確認對象，不存在需要跳過特定師傅的情境
→ 放棄方向：改為逐師傅確認

**移除不新增師傅層級**
→ 一位師傅每期筆數有限，逐筆移除快速，師傅層級移除效益不大
→ 放棄方向：在 Header 加入一次移除全部按鈕

**本次範圍限縮於第三區塊**
→ 避免一次性大幅改動，基於前期優化成果逐步擴增
→ 放棄方向：同步調整三個區塊的師傅分組邏輯

**介面結構選擇 B（側欄＋詳情），放棄 A（手風琴式）**
→ B 的左欄本身即為全師傅總覽，展開明細時版面不位移，操作中仍可保持全局感
→ 放棄方向：手風琴式（展開時版面位移、其他師傅離開視線）
