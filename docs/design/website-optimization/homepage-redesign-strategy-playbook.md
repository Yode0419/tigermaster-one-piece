# 官網首頁改版策略手冊（借鏡 Morph Landing Page V2 方法論）

**日期：** 2026-07-17
**狀態：** 參考文件（方法論 playbook，非單一決策記錄）

---

## 背景

這份文件的方法論來自另一家公司（Morph Technologies）Obsidian 知識庫裡的 Landing Page V2 改版紀錄。那次改版不是「先做視覺稿再討論」，而是用一套固定的 Claude Code skill 分工（`design-strategy` 負責策略生成、`copy-generator` 負責文案迭代、`prototype-builder` 負責原型製作、`ux-advisor` 負責評審、`doc-writer` 負責把每個定案寫回決策記錄）跑完整個流程，並且每個重大轉折都留下「為什麼這樣決定 / 排除了什麼 / 什麼情況要重新考慮」的書面紀錄，讓過程可回溯、可交接，不靠記憶或直覺硬做。

本文件借的是**流程與框架本身**，不是 Morph 的產品內容 —— 目的是給 tigermaster-one-piece 官網首頁改版一套可直接套用的步驟與範本。

**與本專案既有素材的關係：**
- `website-optimization/官網調整1150716.pptx` 與 `website-optimization/官網首頁與關於頁面設計/`（含 `官網改版.dc.html`、`首頁正式版.dc.html` 等原型檔）是既有的改版素材，本文件不取代它們，而是提供一套流程架構，讓後續調整（無論是延續既有原型，還是重新啟動一輪改版）有步驟可循、有紀錄可留。
- 本文件目前是**空白流程模板**，尚未套用到 tigermaster 官網的實際內容（沒有做競品拆解、沒有定 persona、沒有寫決策記錄）。下一步是實際跑過 Phase 0 開始逐步填入。

---

## 整體流程（6 個階段）

### Phase 0｜前期研究基礎
1. **競品拆解分析** — 挑 3-5 個直接競品，systematic 比較色彩系統、版面模式、轉場手法、CTA 密度、UI 呈現方式（真實截圖 vs 純文字 vs 完全沒展示產品）。目標是找出「競品都沒做、但我們做得到」的機會缺口。
2. **目標受眾研究** — 用多來源交叉驗證（既有訪談紀錄、產業指南/白皮書、競品分析、業務團隊回饋、既有產品使用數據），為每個核心受眾建立 persona：人口特徵、心理特徵、痛點、目標、反對理由、決策路徑、視覺偏好。如果有多個受眾，做一份「跨 persona 比較表」找出共通點與差異點。
3.（可選）**品牌視覺角色釐清** — 如果既有品牌識別（logo、主色）已經穩定，先決定「哪些顏色代表什麼角色」（例如：主色=識別、深色=環境背景、強調色=行動呼籲），而不是為了配合新頁面去改動既有識別資產。

### Phase 1｜策略定案（生成式決策，還沒有產出前先想清楚）
1. **敘事骨幹** — 用 Villain（現況痛點）→ Hero（受眾角色）→ Transformation（產品帶來的轉變）三段式，寫出一句核心敘事。
2. **定位聲明** — 每個 persona 一句：「對 [受眾] 而言，[產品] 是唯一 [品類定位] 能做到 [關鍵效益] 的 [方案類型]。」
3. **架構決策** — 至少列出 2 個候選架構（例如：單頁 product-first vs 依受眾分頁），用「優勢/風險」表格逐項比較，並參考競品怎麼做，再定案。
4. **視覺佐證策略** — 決定用什麼證明產品是真的可用：真實截圖、精修 mockup、影片 demo、或混合式。原則是「每個 section 有一個占 60% 以上份量的主視覺」，避免每區都塞一堆小圖分散注意力。
5. **Mood board** — 針對頁面不同區塊（例如 Hero / 功能介紹 / 關於我們）各定一個情緒關鍵字 + 對應色彩配置，讓整頁的情緒節奏是設計過的，不是隨機的。

### Phase 2｜內容生產
1. **文案迭代** — 不要一次寫定，用版本編號（v1 → v2 → v3…）迭代，**每一版都要記錄「這版改了什麼、為什麼改」**，方便回溯哪個決定何時做的。
2. **受眾慣用語稽核** — 把通用行銷術語換成你們產業/受眾真正在用的詞彙。列一份「通用詞 → 產業慣用詞」對照表。

### Phase 3｜原型製作
1. **先判斷設計成熟度，再決定要做什麼**：
   - 方向還沒定 → 做 2-3 個 lofi 變體比較
   - 方向已定但版面還在調 → 沿同方向迭代 lofi
   - 版面穩定了 → 做 hifi，且要涵蓋所有狀態（loading、空狀態、錯誤狀態等）
   - 只是要補一個缺的狀態 → 直接在既有檔案上擴充，不要整份重做
2. **原型要貼合既有產品的真實設計語言**（字體、色票、元件樣式直接從現有 Design System 或既有原型撈，不要憑空發明新樣式），確保官網跟產品本身視覺一致。

### Phase 4｜評審與定案
1. 到達一個「lock point」（版面/文案都相對穩定）時，做一輪 heuristic 審查 —— 找沒有參與製作過程的人用「新鮮眼睛」看，抓可用性問題、或在兩個候選方案間當 tiebreaker。
2. 明確標記哪個版本是「目前定案版」，避免多個版本並存造成混淆。

### Phase 5｜工程交接
1. 把 prototype 畫不出來的東西寫成規格：所有互動狀態、邊界案例、資料規則、響應式斷點行為。

### Phase 6｜決策留存（貫穿全程，不是最後才做）
1. 每個重大決策都寫一份「決策記錄」（格式見下方範本 D），存放位置比照本 repo 既有慣例（`docs/design/` 下的 `.md` 決策記錄樣式，可參考同目錄的 [`frame-size-standard.md`](../frame-size-standard/frame-size-standard.md)）。
2. **保持「總覽/README」與實際進度同步更新**。查考來源專案時發現，對方的專案總覽文件還寫著「目前是 v7 版、暫停中」，但實際原型早就進到 v11 版且已經在跑額外的同步工具 —— 這種「總覽文件」和「實際產出」脫節的狀況很容易發生。建議每完成一個里程碑就順手更新總覽，不要等到事後才補。
